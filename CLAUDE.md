# Markdown Org -- правила проекта

## Покрытие изменений тестами

Любое изменение поведения кода ОБЯЗАТЕЛЬНО должно быть покрыто тестами
(unit или интеграционными). Это относится и к фиксам багов, и к новым
функциям, и к рефакторингам, где меняется наблюдаемое поведение.

Из этого следует:

- Если для изменяемого кода уже есть тест -- проверить, что он покрывает
  новое поведение; при необходимости расширить или добавить новый case.
- Если тестов на этот участок ещё не было -- завести их в той же ветке,
  что и сами изменения. Не откладывать "на потом".
- Чистые refactor без изменения поведения (переименование, перенос
  кода) могут опираться на существующие тесты; всё равно нужно убедиться,
  что они прогоняются после изменения.
- Unit-тесты живут в `src/test/**/*.test.ts`, интеграционные --
  `src/test/**/*.integration.test.ts`. Запуск: `npm run test` и
  `npm run test:integration` соответственно.
- В PR/commit не должно быть ситуации "поправил баг -- тестов нет".
  В commit message можно сослаться на добавленный тест по имени.

## Документация фич в README

При появлении новой пользовательской фичи ИЛИ изменении наблюдаемого
поведения существующей ОБЯЗАТЕЛЬНО обновлять раздел `## Features` в
`README.md` в той же ветке, что и сам код. README -- это первое, что
видит пользователь в VS Code Marketplace / Open VSX, и он должен
отражать реальный набор возможностей расширения.

Из этого следует:

- Новая фича -- добавить отдельный пункт (или подсекцию, если фича
  крупная и заслуживает выделения) в `## Features`. Крупную фичу с
  собственной детальной секцией (как `## Google Calendar Sync`)
  выделять подсекцией внутри Features со ссылкой `#` на детальную
  секцию и, по возможности, ведущим демо-GIF.
- Изменение поведения существующей фичи -- обновить её пункт в
  Features, чтобы формулировка соответствовала новому поведению (новые
  ограничения, новые опции, изменённые дефолты).
- Удаление фичи -- убрать соответствующий пункт.
- TOC (`## Table of Contents`) поддерживать в синхроне с заголовками.
- Обновление README идёт в том же коммите, что и код фичи (или в
  ближайшем последующем `docs(...)`-коммите той же ветки); не
  откладывать "на релиз".
- Если фича невидима пользователю (только внутренний рефакторинг,
  changes без API/UI/keybinding/settings) -- README не трогать.
- Признак "наблюдаемого поведения", требующий правки README: новая или
  переименованная команда, новый keybinding, новая/изменённая
  настройка `markdown-org.*`, новый формат вывода, новое окно/панель,
  изменённый дефолт. Если сомневаетесь -- считать, что README обновить
  надо.

CHANGELOG обновляется отдельно по своим правилам (Keep a Changelog);
README отражает текущее состояние, CHANGELOG -- историю.

## Extractor и навигация по задачам

`markdown-org-extract` -- это инструмент широкого поиска: он сканирует
произвольные файлы и сводит задачи в единую agenda. Семантика расширения --
показать всё, что нашёл extractor, и позволить перейти к найденному.

Из этого следует:

- Не добавлять проверок вида `isPathInsideWorkspace` на пути, полученные
  от extractor (handler `openTask` в `src/views/agendaPanel.ts`). Если
  extractor вернул файл -- его обязательно надо суметь открыть, даже если
  он лежит вне `vscode.workspace.workspaceFolders` (симлинки, отдельный
  `markdown-org.workspaceDir`, агрегация из нескольких каталогов).
- Проверка `isPathInsideWorkspace` остаётся уместной для пользовательских
  настроек, валидируемых до записи (`markdown-org.maintainFilePath` в
  `src/commands/moveHeading.ts`) -- это другой сценарий, там нужна защита
  от случайной записи в системные пути.

## Безопасность webview: CSP + escapeHtml

Все webview-панели (`src/views/agendaPanel.ts`, `src/views/clocktablePanel.ts`
и любые будущие) ОБЯЗАНЫ соблюдать инвариант:

- В HTML присутствует `<meta http-equiv="Content-Security-Policy">` с
  `default-src 'none'`, `style-src ${cspSource} 'nonce-${nonce}'`,
  `script-src 'nonce-${nonce}'`. Никаких `unsafe-inline`, `unsafe-eval`,
  `*` или внешних источников без явного обоснования.
- Все `<style>` и `<script>` теги несут атрибут `nonce="${nonce}"`,
  совпадающий с CSP. Nonce генерируется на каждое создание панели
  (`generateNonce()`), не переиспользуется.
- Любые данные, попадающие в innerHTML/строки HTML на стороне расширения
  и любые данные task'ов (`file`, `heading`, `tags`, `status`, `priority`,
  `timestamp_time`, заголовки дней и т.д.), ОБЯЗАНЫ проходить через
  `escapeHtml` перед вставкой. Числовые поля (`task.line`) санитизируются
  как число (`sanitizeTaskLine` или `Number(...)`), а не как строка.
- В обработчике `webview.onDidReceiveMessage` каждое поле сообщения
  проверяется по типу (`typeof message.file !== 'string'`,
  `typeof message.line !== 'number'`) ДО передачи в команды VS Code или
  файловые операции. Сообщения с непроверенными полями обрабатывать
  нельзя.
- Команда `openTask` -- единственный путь открытия файлов из webview;
  она использует `vscode.workspace.openTextDocument(file)`, файл при этом
  не ограничивается границами workspace (см. секцию "Extractor и навигация
  по задачам"), но строковый тип `file` обязателен.

Из этого следует:

- При добавлении нового поля задачи, выводимого в webview, сразу же
  оборачивать его в `escapeHtml` (для строк) или `sanitizeTaskLine`/числовое
  приведение (для чисел). Не "потом, когда станет видно XSS".
- При добавлении нового сообщения от webview к расширению писать
  явный type guard на каждое поле перед использованием.
- При расширении CSP (новые источники стилей, шрифтов, изображений)
  обновлять и сам тег CSP, и комментарий с обоснованием, что именно
  потребовалось разрешить и почему.
- Изменения в webview покрываются тестами (см. секцию "Покрытие изменений
  тестами"); для CSP/escapeHtml достаточно integration-теста, проверяющего,
  что в сгенерированном HTML есть нужный CSP-тег и нет user-controlled
  строк без экранирования.

## Сериализация JSON: safe-stable-stringify

В штатном (не-тестовом) коде расширения ЗАПРЕЩЕНО использовать
`JSON.stringify` для значений, которые попадают в один из следующих
сценариев:

- запись в файл, в settings или в любое другое хранилище, где
  потом будет сравнение/diff'ы по содержимому;
- логирование/отчёты, которые читает человек или сторонняя система
  и где порядок ключей и отсутствие цикла важны;
- сериализация значений из задач/extractor, где могут встретиться
  `BigInt`, циклические ссылки или объекты с нестабильным порядком
  ключей.

В таких местах использовать пакет
[`safe-stable-stringify`](https://github.com/BridgeAR/safe-stable-stringify):
он даёт стабильную сортировку ключей, безопасно обрабатывает циклы
(`[Circular]` вместо TypeError) и `BigInt` (как число вместо
исключения), API совпадает с `JSON.stringify(value, replacer?, space?)`.

Из этого следует:

- При первом появлении такого вызова в штатном коде добавить
  `safe-stable-stringify` в `dependencies` и импортировать его как
  `import stringify from 'safe-stable-stringify'` (через
  `esModuleInterop`).
- `JSON.stringify` остаётся допустимым в тестах и demo-инфраструктуре
  (`src/test/**`, `scripts/**`), где порядок ключей не наблюдаем
  снаружи: моки `exec`-вывода extractor, маркер `markDemoStart`,
  диагностические `console.log` и т.п.
- На момент написания правила в `src/**` (кроме `src/test/**`)
  вызовов `JSON.stringify` нет; правило носит превентивный характер.
